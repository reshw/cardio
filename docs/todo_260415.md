### 1. 카카오 공유 URL 수정 (`WorkoutFeedCard.tsx`)

**기존**: 공유 버튼 URL이 `/club` (고정)으로, feedId/clubId 정보 없음
**변경**: `/workout/{workoutId}?clubId={clubId}` 로 수정

```
기존: ${window.location.origin}/club
변경: ${window.location.origin}/workout/${workout.id}?clubId=${clubId}
```

---

### 2. 로그인 후 복귀 URL 보존 (`KakaoLogin.tsx`)

**기존**: `window.location.pathname` 만 state에 저장 → 쿼리 파라미터 유실
**변경**: `window.location.pathname + window.location.search` 로 수정

로그인 후 KakaoCallback에서 `?clubId=...` 파라미터까지 포함된 URL로 복귀 가능해짐.

---

### 3. 클럽 소속 여부 체크 및 join 리디렉트 (`WorkoutDetail.tsx`)

URL에 `clubId` 파라미터가 있을 때 아래 순서로 처리:

1. `authLoading` 완료 대기 (auth 준비 전 실행 방지 - 핵심)
2. `clubService.isClubMember(clubId, user.id)` 로 소속 여부 확인
3. 비가입 시: `clubService.getClubById(clubId)` → `invite_code` 조회
4. `join_return_url` = `/workout/{id}?clubId={clubId}` 를 sessionStorage에 저장
5. `/join/{invite_code}` 로 navigate

**중요**: `clubIdParam` 이 있을 때 멤버십 체크를 워크아웃 로드보다 **먼저** 실행.
기존에는 두 작업이 동시에 실행되어, RLS로 워크아웃 조회가 차단되면 `navigate('/')` 가 join 리디렉트보다 먼저 실행되는 race condition 발생.

---

### 4. 가입 완료 후 피드 복귀 (`JoinClub.tsx`)

가입 성공 / 이미 가입된 경우 모두 `join_return_url` 확인:
- 있으면 → 해당 피드 URL로 복귀
- 없으면 → 기존처럼 `/club` 으로 이동

## 버그 히스토리

### 버그 1: 공유 링크 클릭 시 History 페이지로 이동
**원인**: 두 가지 race condition
1. `WorkoutFeedCard`의 공유 URL이 `/club` 이라 WorkoutDetail이 실행되지 않음
2. `loadWorkout()` 과 `checkClubMembership()` 이 병렬 실행 → RLS로 워크아웃 조회 실패 시 `navigate('/')` 가 먼저 실행

**해결**: 공유 URL에 workoutId+clubId 포함, 멤버십 체크 → 워크아웃 로드 순서 직렬화

---

### 버그 2: 공유 링크 클릭 시 `/club` 페이지로 이동
**원인**: Supabase auth 타이밍 race condition
- Kakao 로그인 직후, auth 토큰이 Supabase 클라이언트에 완전히 전파되기 전에 `isClubMember` 실행
- WorkoutDetail: auth 미준비 상태 → `isClubMember` false 반환 → 비가입자로 판단 → `/join` 이동
- JoinClub: auth 완전 준비된 시점에 다시 `isClubMember` 실행 → true 반환 → `/club` 이동

**해결**:
1. WorkoutDetail에 `authLoading` 체크 추가 → auth 완전 준비 후에만 멤버십 체크 실행
2. JoinClub `loadClubPreview`의 "이미 가입" 분기에서도 `join_return_url` 확인 후 복귀 (안전망)
