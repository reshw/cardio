# 패치노트 & 업데이트 소통 방안

---

## 현재 상황 분석

### 구조적 위치
- **나**: 앱 기획자 + 개발자 (기술적 권한 100%)
- **동호회 운영진**: 현재 서비스 오너 (기능 방향 결정권)
- **일반 유저**: 동호회 회원 (앱의 실사용자)

### 불편함의 본질
기술을 만든 사람과 서비스를 소유한 사람이 분리된 구조다.
흔히 "외주 개발자"처럼 보일 수 있지만, 실제로는 그보다 훨씬 깊이 관여되어 있다.
운영진이 나를 존중한다는 건 좋은 신호지만, 그 존중이 **명확한 역할 정의** 없이 유지되면 언제든 모호해질 수 있다.

핵심 긴장 포인트:
- 내가 패치를 내면 "운영진의 결정"인가, "개발자의 결정"인가?
- 유저 입장에서 누구한테 피드백을 줘야 하는가?
- 기능 개선이 운영진의 공로로 소비될 때 개발자의 존재감은?

---

## 소통 방안 제안

### 1. 패치노트 발행 주체를 명확히 구분

**권장 구조:**

```
[운영진 채널] → 공식 공지 (기능 추가/변경 사실 전달)
[개발자 채널] → 기술적 배경 설명, 버그 수정, 개선 의도 공유
```

운영진이 "이런 기능이 생겼어요"를 공지하면,
나는 "왜 이렇게 만들었는지"를 보조 설명하는 구조.

이렇게 하면:
- 운영진의 오너십이 유지됨 (공지 주체)
- 나의 기술적 권위도 살아있음 (설계 의도 설명)
- 유저는 두 채널을 통해 더 풍부한 정보를 얻음

---

### 2. 패치노트 형식 — 두 버전 준비

#### 버전 A: 운영진용 (공지 템플릿)
> 간결하고 사용자 친화적. 운영진이 카톡방 등에 바로 붙여넣을 수 있게.

```
📱 앱 업데이트 안내

✅ 이번 달 통계를 월별로 볼 수 있게 됐어요
✅ 클럽 순위 화면이 더 보기 좋아졌어요
✅ 운동 등록 순서 표시 방식 개선

불편하신 점은 언제든 말씀해 주세요 🙏
```

#### 버전 B: 개발자 채널용 (심층 설명)
> 좀 더 기술적, 의도 중심. 관심 있는 유저나 운영진이 깊이 읽을 수 있게.

```
이번 업데이트에서 "n번째 운동" 표시 기준을 바꿨습니다.

기존엔 운동한 시각(workout_time) 기준이라,
저녁에 오전 운동을 늦게 올리면 번호가 뒤섞이는 문제가 있었어요.
이제는 등록한 순서(created_at) 기준으로 번호가 매겨져서
한번 공유된 "17번째"는 다음날도 17번째로 유지됩니다.
```

---

### 3. 역할 명칭 정리 (내부적으로라도)

모호한 상태를 그냥 두면 나중에 의사결정 충돌이 생긴다.
지금 당장 공식화하지 않더라도, 운영진과 나 사이에서만큼은 아래 구분을 공유해두는 게 좋다.

| 영역 | 결정권자 |
|------|---------|
| 기능 기획 (무엇을 만들까) | 운영진 → 나 순으로 논의 |
| 기술 구현 방식 (어떻게 만들까) | 나 단독 |
| 배포 타이밍 | 나 (운영진 동의 후) |
| 유저 공지 | 운영진 주도, 나 보조 |
| 버그/긴급 수정 | 나 단독 (사후 공유) |

---

### 4. 소통 빈도 제안

| 주기 | 내용 |
|------|------|
| 수시 | 버그 수정, 긴급 패치 → 카톡 한 줄 공유 |
| 2주~1달 | 기능 업데이트 패치노트 발행 |
| 분기 | 로드맵 공유 + 운영진과 방향 논의 |

---

## 핵심 원칙

**투명하되, 주도권을 뺏지 않는다.**

운영진이 앱을 자기 것처럼 느낄수록 앱에 더 애착을 갖고 유저들에게 더 잘 전달한다.
내가 전면에 나서는 것보다, 운영진이 "우리 앱이 이렇게 좋아졌어요"라고 자랑하게 만드는 게 장기적으로 더 낫다.

개발자의 존재감은 패치노트의 전면이 아니라, **퀄리티**와 **설명의 깊이**로 드러내면 된다.
